# TARA Matrix™ v3.0 — FASE 3
## Release oficial · STABLE · 29 de junio de 2026

---

## Objetivo de la fase

Transformar TARA de un asistente single-tenant (hardcodeado para Total Racks) en una plataforma multi-tenant real, capaz de servir a múltiples empresas desde un solo servidor sin ningún tipo de contaminación de datos, personalidad ni comportamiento entre ellas.

## Cambios principales

### Arquitectura de routing (el cambio más importante)

**Antes (v2.x — single-tenant):**
```
Mensaje entra → COMPANY_SLUG (variable de entorno) → config de una sola empresa
```

**Después (v3.0 — multi-tenant):**
```
Mensaje entra → incoming_endpoint (número receptor Twilio)
             → channel_endpoints (tabla Supabase)
             → company_id
             → obtenerConfigEmpresa(company_id)
             → config de la empresa correcta
```

### Flujo completo del webhook (T3.9)
```js
const message         = adapter.parseIncoming(req);        // incluye incoming_endpoint
const routeResult     = await channelRouter.enrutar(message.incoming_endpoint);
if (!routeResult) return res.type('text/xml').send('<Response></Response>');
message.company_id    = routeResult.company_id;
const resultado       = await orchestrator.procesarMensaje(message);
```

### Archivos modificados
- `server.js` — routing activo, sin COMPANY_SLUG
- `modules/config.js` — `obtenerConfigEmpresa(companyId)`, Map cache
- `modules/crm.js` — todas las funciones reciben y filtran por `company_id`
- `modules/orchestrator.js` — 4 líneas ajustadas para pasar `company_id`
- `adapters/channels/twilio-whatsapp.js` — expone `incoming_endpoint`
- `render.yaml` — COMPANY_SLUG eliminado
- `.env.example` — COMPANY_SLUG eliminado

### Archivos nuevos

## Arquitectura alcanzada

```
Twilio WhatsApp
      │
      ▼ POST /webhook/twilio
┌─────────────────────────────────────────────────────┐
│  server.js                                          │
│                                                     │
│  TwilioWhatsAppAdapter.parseIncoming()              │
│        │ incoming_endpoint: "whatsapp:+1415..."     │
│        ▼                                            │
│  ChannelRouter.enrutar(endpoint)                    │
│        │ → channel_endpoints (Supabase)             │
│        │ → { company_id, company_slug }             │
│        ▼                                            │
│  message.company_id = company_id                    │
│        │                                            │
│        ▼                                            │
│  Orchestrator.procesarMensaje(message)              │
│        │                                            │
│        ├── obtenerConfigEmpresa(company_id)         │
│        ├── obtenerOCrearCliente(tel, company_id)    │
│        ├── guardarConversacion(..., company_id)     │
│        └── crearOportunidad(..., company_id)        │
└─────────────────────────────────────────────────────┘
```

**Invariante de aislamiento:** `company_id` fluye en una sola dirección — desde el número receptor → hasta cada escritura en DB. Nunca se mezcla entre empresas.

## Riesgos resueltos

| Riesgo | Mitigación aplicada |
|--------|---------------------|
| Corte en producción (T3.9) | Total Racks seeded en channel_endpoints desde T3.2 — antes del corte |
| Datos históricos sin company_id | Backfill automático en T3.3 con UUID de Total Racks |
| Cache de config incorrecta | Map cache por company_id — nunca sirve config de otra empresa |
| Crash por número desconocido | Channel Router devuelve null → TwiML vacío, sin crash |
| Rollback sin punto de retorno | Git tag `fase-3-inicio` + ROLLBACK-FASE3.md con SQL reversible |

## Qué quedó fuera deliberadamente

| Item | Razón |
|------|-------|
| FASE 3.5 (company_id separado) | Fusionado en FASE 3 por decisión de la fundadora — evitar deuda técnica |
| UI de onboarding para nuevas empresas | Fuera del alcance técnico de FASE 3 — se hace por SQL directo en Supabase |
| Validación de firma Twilio por empresa | Un solo Auth Token es suficiente para el sandbox; producción multi-número es FASE 4+ |
| RLS (Row Level Security) en Supabase | Aislamiento garantizado por código — RLS añade latencia sin beneficio en backend server-side |
| Backup automático pre-FASE3 | Script disponible, pero requiere credenciales que solo existen en Render |
| Segundo canal (SMS, Instagram) | La arquitectura lo soporta (`canal` en channel_endpoints), pero no se implementó |

## Lecciones aprendidas

**1. "El corte" no fue el momento más riesgoso.**
El riesgo real era tener datos históricos sin `company_id`. El backfill de T3.3 eliminó ese riesgo antes de T3.9. Para FASE 4: identificar el equivalente al "backfill" y hacerlo primero.

**2. La secuencia de tareas importa más que la velocidad.**
T3.2 (seed del número) antes de T3.9 (el corte) fue la decisión correcta. Si T3.9 se hubiera hecho antes, Total Racks habría dejado de responder hasta que T3.2 se completara.

**3. Los tests de fixture deben ser coherentes con los mocks.**
`makeMessage()` con `company_id: null` pasó durante semanas. El error emergió al agregar `company_id` a las firmas de CRM. Regla: cuando cambia una firma, revisar los fixtures de prueba antes de los mocks.

**4. La arquitectura hexagonal pagó dividendos.**
Los 4 cambios en `orchestrator.js` fueron de una línea cada uno. El Kernel nunca sabe de Twilio ni de OpenAI. Agregar una empresa nueva es 4 inserts en Supabase, sin tocar código.

**5. Documentar decisiones NO tomadas es tan valioso como documentar las tomadas.**
El README de FASE 3 incluye una sección "Decisiones no tomadas" que evitará re-debatir lo mismo en FASE 4.
