# T3.8 — Refactor mínimo `orchestrator.js`

**Estado:** Pendiente  
**Tipo:** Código  
**Archivo:** `modules/orchestrator.js`  
**Riesgo:** Alto — el Orchestrator es el Kernel. Cualquier cambio aquí afecta 100% de los mensajes procesados.

---

## Objetivo

El Orchestrator recibe un `Message` que ya tiene `company_id` resuelto (asignado por `server.js` después de T3.9). Necesita pasar ese `company_id` a las dos funciones que lo necesitan: `_obtenerConfig` y `_obtenerCliente`.

Este es intencionalmente el cambio más pequeño de toda FASE 3.

---

## Por qué el cambio debe ser mínimo

El Orchestrator es Kernel. El Artículo 15 de la Constitución dice: "El Kernel nunca debe contaminarse con lógica de negocio." Agregar routing logic, queries de DB o conocimiento sobre empresas al Orchestrator violaría este principio.

El Orchestrator no "sabe" que el sistema es multi-tenant. Solo sabe que recibió un `Message` que tiene un `company_id`, y ese `company_id` se pasa como dato a las funciones que lo necesitan. La responsabilidad de resolver el `company_id` es del Channel Router y del server.js, no del Kernel.

---

## Los dos cambios

**Cambio 1 — Paso de config:**
```js
// Antes (línea 116 actual):
const configResult = await this._paso('config', timings, () =>
  this._obtenerConfig()
);

// Después:
const configResult = await this._paso('config', timings, () =>
  this._obtenerConfig(message.company_id)
);
```

**Cambio 2 — Paso de CRM:**
```js
// Antes:
const clienteResult = await this._paso('crm', timings, () =>
  this._obtenerCliente(message.from)
);

// Después:
const clienteResult = await this._paso('crm', timings, () =>
  this._obtenerCliente(message.from, message.company_id)
);
```

**Cambio 3 — guardarConversacion (en la sección de guardado):**
```js
// Buscar en orchestrator.js donde se llama _guardarConv y agregar company_id
// El company_id ya está disponible como variable `company_id` en ese scope
this._guardarConv(clienteRaw?.id, message.company_id, message.content, ...)
```

Eso es todo. Tres cambios de una línea cada uno. Ninguna lógica nueva, ningún import nuevo.

---

## Garantía de que los tests siguen pasando

Los tests del Orchestrator usan funciones inyectadas (el Orchestrator tiene DI completo). Los tests que pasan `obtenerConfigEmpresa` como mock no necesitan cambiar — el mock simplemente recibirá un parámetro que puede ignorar:

```js
// Test existente (sigue funcionando):
const mockConfig = async () => ({ company: {...}, personality: {...}, knowledge: [] });

// Test actualizado (más correcto):
const mockConfig = async (companyId) => ({ company: { id: companyId, ... }, ... });
```

Si los tests existentes no pasan `company_id` en el Message, agregarle el campo `company_id: 'test-id'` a los fixtures de Message.

## Notas

**T3.6 + T3.7 + T3.8 deben aplicarse en la misma sesión** sin deploy entre ellos. El deploy solo ocurre en T3.9, cuando todo el pipeline está conectado y probado localmente. Hacer un deploy parcial (por ejemplo, solo T3.6) causaría que `server.js` llame a `_obtenerConfig()` sin parámetro, lo cual falla con el nuevo `config.js`.

Esta es la razón por la que T3.9 es "el corte": es el primer momento donde el sistema completo, con todos los cambios, entra a producción como una unidad.
