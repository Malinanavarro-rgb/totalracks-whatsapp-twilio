## Objetivo

Crear el módulo que responde la pregunta central del routing multi-tenant: dado un `incoming_endpoint` (número de Twilio receptor), ¿a qué empresa pertenece?

---

## Por qué es un módulo separado y no lógica en server.js

`server.js` ya tiene demasiadas responsabilidades: recibir webhooks, validar firmas, coordinar el flujo y formatear respuestas. Si además hace queries de routing, violaría el principio de responsabilidad única.

Separar el router en su propio módulo también permite:
- Testearlo de forma aislada sin levantar el servidor
- Cachear el routing independientemente de la config de empresa
- Reemplazarlo en el futuro (ej: routing por header HTTP en lugar de número)

---

## Interfaz del módulo

```js
const router = new ChannelRouter(supabase);

// Resolver endpoint a empresa
const resultado = await router.enrutar('whatsapp:+14155238886');
// → { company_id: 'uuid...', company_slug: 'totalracks' }
// → null si el endpoint no está registrado o está inactivo
```

---

## Lógica de caché

El caché es un `Map<endpoint, { company_id, company_slug, cachedAt }>` con TTL de 5 minutos.

Por qué 5 minutos: es el mismo TTL que usa `config.js`. Si se actualiza un endpoint en Supabase, el cambio tarda máximo 5 minutos en ser visible, o se puede forzar reiniciando el servidor.

Por qué `Map` y no una variable global: permite que el router sea una instancia con estado controlado, más fácil de testear y de resetear sin reiniciar el servidor (ej: `router.invalidarCache(endpoint)` en una futura API de administración).

---

## Comportamiento ante errores

- **Endpoint no registrado:** devuelve `null`. El servidor responde con TwiML vacío.
- **Endpoint inactivo (`activo = false`):** devuelve `null`. Mismo comportamiento.
- **Error de DB:** devuelve `null` + `console.error`. El servidor responde con TwiML vacío. No lanza excepción.
- **Cache hit:** devuelve resultado en memoria sin tocar Supabase.

La decisión de devolver `null` en lugar de lanzar error es coherente con el Principio P7 de la Constitución (Fail gracefully, siempre): un número desconocido no es un error fatal, es una condición esperada.

## Prueba en aislamiento

```bash
# Desde la raíz del repo, con .env configurado:
node -e "
require('dotenv').config();
const { createClient } = require('@supabase/supabase-js');
const { ChannelRouter } = require('./modules/channel-router');
const sb = createClient(process.env.SUPABASE_URL, process.env.SUPABASE_ANON_KEY);
const router = new ChannelRouter(sb);
router.enrutar('whatsapp:+14155238886').then(r => console.log('Resultado:', r));
"
```

Resultado esperado: `{ company_id: 'uuid...', company_slug: 'totalracks' }`

Segunda prueba — número inexistente:
```bash
node -e "

"
```

Resultado esperado: `null`
