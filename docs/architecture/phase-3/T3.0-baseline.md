## Objetivo

Crear un punto de restauración completo antes de tocar cualquier archivo o tabla. Si algo sale mal en cualquier tarea posterior, este baseline permite volver al estado exacto de FASE 2 en menos de 5 minutos.

---

## Por qué existe esta tarea

Las migraciones de base de datos son las únicas operaciones verdaderamente difíciles de deshacer en este stack. Un `git revert` restaura el código en segundos. Restaurar datos en Supabase sin un backup previo puede ser imposible si hay datos de producción comprometidos.

La regla es: nunca empezar una migración sin poder terminarla o deshacerla por completo.

Esta tarea también sirve como fotografía del estado de FASE 2 para el registro histórico. En seis meses, será útil saber exactamente qué existía antes de FASE 3.

### T3.0b — Backup del esquema de Supabase

Ejecutar en Supabase SQL Editor → guardar resultado como `backups/pre-fase3-schema.sql`:

```sql
SELECT
  table_name,
  column_name,
  data_type,
  is_nullable,
  column_default
FROM information_schema.columns
WHERE table_schema = 'public'
  AND table_name IN (
    'companies', 'personalities', 'knowledge_base', 'decision_logs',
    'clientes', 'conversaciones', 'oportunidades', 'cotizaciones'
  )
ORDER BY table_name, ordinal_position;
```

### T3.0c — Backup de datos de configuración actuales

Ejecutar en Supabase SQL Editor → guardar resultado como `backups/pre-fase3-data.txt`:

```sql
SELECT 'COMPANIES' AS seccion, id::text, slug, nombre, estado, created_at::text
FROM companies
UNION ALL
SELECT 'PERSONALITIES', id::text, company_id::text, nombre_asistente, tono, created_at::text
FROM personalities
UNION ALL
SELECT 'KNOWLEDGE_BASE', id::text, company_id::text, categoria, LEFT(contenido, 100), created_at::text
```
