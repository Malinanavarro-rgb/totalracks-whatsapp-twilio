# T3.4 — Exponer `incoming_endpoint` en el Adapter

**Estado:** Pendiente  
**Tipo:** Código  
**Archivo:** `adapters/channels/twilio-whatsapp.js`  
**Riesgo:** Bajo — cambio aditivo. No elimina ni modifica campos existentes del objeto Message.

---

## Objetivo

El `TwilioWhatsAppAdapter` actualmente lee `req.body.From` (quién envía) pero ignora `req.body.To` (a qué número llegó el mensaje). `req.body.To` es la única forma de saber a cuál de las empresas registradas pertenece este mensaje entrante.

Esta tarea agrega `incoming_endpoint` al objeto Message normalizado.

---

## Por qué lo lee el Adapter y no el server.js

El Artículo 12 de la Constitución dice: "El Channel Adapter es el único que conoce el exterior." El formato de `req.body.To` es un detalle de Twilio (`whatsapp:+521XXXXXXXXXX`). Nadie más debería leer directamente el body del request.

`server.js` no debe hacer `req.body.To`. Debe obtenerlo del objeto Message ya normalizado: `message.incoming_endpoint`.

Esto garantiza que cuando se agregue un adapter de Instagram, ese adapter también exponga `incoming_endpoint` de la manera que Instagram lo proporcione — y el Channel Router funciona igual para ambos sin cambiar.

---

## Cambio

**Antes** (objeto Message retornado por `parseIncoming`):
```js
{
  id, company_id: null, channel, from, content, timestamp, raw_metadata
}
```

**Después:**
```js
{
  id,
  company_id:        null,              // el Channel Router lo asigna en server.js
  channel:           this.canal,
  from,
  incoming_endpoint: req.body.To || null,  // nuevo — formato: "whatsapp:+521XXXXXXXXXX"
  content,
  timestamp,
  raw_metadata: { ... }
}
```

`incoming_endpoint` lleva el valor crudo de Twilio, con el prefijo `whatsapp:`. El Channel Router lo usará exactamente así para buscar en `channel_endpoints`.

## Verificación


Enviar un mensaje de prueba por WhatsApp → verificar en logs de Render que aparece el valor esperado (ej: `whatsapp:+14155238886`).

Eliminar el `console.log` antes del commit final de esta tarea.
