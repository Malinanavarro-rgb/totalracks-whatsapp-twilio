# T3.3 — `company_id` en tablas CRM

**Estado:** Pendiente  
**Tipo:** DB migration  
**Archivo:** `migrations/003_company_id_en_crm.sql`  
**Riesgo:** Medio — modifica tablas con datos existentes. Requiere backfill correcto.

---

## Objetivo

Agregar `company_id` a las tablas operativas del CRM (`clientes`, `conversaciones`, `oportunidades`) para que cada fila pertenezca explícitamente a una empresa.

---

## Por qué es parte de FASE 3 y no una fase posterior

Sin esta columna, el routing funciona (el mensaje llega a la empresa correcta) pero el aislamiento de datos no. Dos empresas distintas podrían tener contactos con el mismo número de teléfono y compartirían la misma fila en `clientes`. El historial de conversaciones de una empresa podría "colarse" en la otra.

El Artículo 8 de la Constitución TARA Matrix dice: "Los Contacts son de Company." Eso no es posible si la tabla `clientes` no tiene `company_id`. FASE 3 no puede considerarse "multiempresa real" sin este aislamiento.

---

## Schema actual (antes)

```
clientes:       id, telefono, nombre, empresa, ciudad, email, fuente, estado, score_interes, ...
conversaciones: id, cliente_id, mensaje_cliente, respuesta_tara, ...
oportunidades:  id, cliente_id, tipo_rack, estado, presupuesto_estimado, ...
```

## Schema objetivo (después)

```
clientes:       id, company_id, telefono, nombre, ... (company_id nuevo)
conversaciones: id, company_id, cliente_id, ...       (company_id nuevo, redundante pero necesario para queries directas)
oportunidades:  id, company_id, cliente_id, ...       (company_id nuevo)
```

**¿Por qué `company_id` en `conversaciones` si ya está en `clientes`?**  
Para queries de dashboard y reporting sin necesidad de JOIN. "Dame todas las conversaciones de la empresa X en los últimos 7 días" no debería requerir pasar por `clientes`. La redundancia es intencional y es un patrón aceptado en arquitecturas multi-tenant.

---

## SQL de migración

```sql
-- ── clientes ──────────────────────────────────────────────────────────────────
ALTER TABLE clientes
  ADD COLUMN company_id uuid REFERENCES companies(id);

-- Backfill: todos los clientes actuales son de Total Racks
UPDATE clientes
SET company_id = (SELECT id FROM companies WHERE slug = 'totalracks')
WHERE company_id IS NULL;

-- Índice compuesto para queries multi-tenant
CREATE INDEX idx_clientes_company_telefono
  ON clientes (company_id, telefono);

-- ── conversaciones ────────────────────────────────────────────────────────────
ALTER TABLE conversaciones
  ADD COLUMN company_id uuid REFERENCES companies(id);

UPDATE conversaciones
SET company_id = (SELECT id FROM companies WHERE slug = 'totalracks')
WHERE company_id IS NULL;

CREATE INDEX idx_conversaciones_company_time
  ON conversaciones (company_id, created_at DESC);

-- ── oportunidades ─────────────────────────────────────────────────────────────
ALTER TABLE oportunidades
  ADD COLUMN company_id uuid REFERENCES companies(id);

UPDATE oportunidades
SET company_id = (SELECT id FROM companies WHERE slug = 'totalracks')
WHERE company_id IS NULL;

CREATE INDEX idx_oportunidades_company_estado
  ON oportunidades (company_id, estado);
```

**Nota:** Las columnas se dejan nullable inicialmente (`NOT NULL` se agrega en FASE 4 cuando todos los flujos de escritura estén actualizados). Durante FASE 3, el código de `crm.js` siempre pasará el `company_id`, pero se evita una restricción que podría romper código antiguo no actualizado.

## Verificación post-ejecución

```sql
-- Verificar columnas existen
SELECT table_name, column_name
FROM information_schema.columns
WHERE table_name IN ('clientes', 'conversaciones', 'oportunidades')
  AND column_name = 'company_id';
-- Esperado: 3 filas

-- Verificar que el backfill funcionó
SELECT
  'clientes'       AS tabla, COUNT(*) AS total, COUNT(company_id) AS con_company_id FROM clientes
UNION ALL
SELECT
  'conversaciones', COUNT(*), COUNT(company_id)                                     FROM conversaciones
UNION ALL
SELECT
  'oportunidades',  COUNT(*), COUNT(company_id)                                     FROM oportunidades;
-- Esperado: total = con_company_id en cada fila (cero NULLs después del backfill)
```

## Rollback

```sql
ALTER TABLE clientes       DROP COLUMN IF EXISTS company_id;
ALTER TABLE conversaciones DROP COLUMN IF EXISTS company_id;
ALTER TABLE oportunidades  DROP COLUMN IF EXISTS company_id;
```

Esto no elimina datos de clientes/conversaciones/oportunidades — solo elimina la columna. Los datos de Total Racks sobreviven intactos.

---

## Notas

Si hay muchas filas en estas tablas (miles), el backfill puede tardar. En Supabase Free tier con datos de prueba, debería ser instantáneo.

La decisión de dejar `company_id` como nullable y no hacer `ALTER TABLE ... ALTER COLUMN company_id SET NOT NULL` hasta FASE 4 es conservadora pero correcta: si hay algún código que aún no pasa el `company_id`, mejor que inserte un NULL (visible, auditable) que que falle silenciosamente.
