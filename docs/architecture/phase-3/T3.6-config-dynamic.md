# T3.6 — Refactor `config.js` — Cache dinámico por `companyId`

**Estado:** Pendiente  
**Tipo:** Código  
**Archivo:** `modules/config.js`  
**Riesgo:** Medio — este módulo es crítico. Un error aquí impide cargar config para cualquier empresa.

---

## Objetivo

Transformar `obtenerConfigEmpresa()` de una función que retorna la config de una empresa fija (determinada por `COMPANY_SLUG`) a una función que retorna la config de cualquier empresa dado su UUID.

---

## Por qué el cambio es necesario aquí

`config.js` es el último módulo que todavía tiene `COMPANY_SLUG` quemado en su lógica. Tiene un único cache de valor `_configCache` que solo puede contener la configuración de una empresa a la vez.

Si dos empresas llaman a `obtenerConfigEmpresa()` al mismo tiempo, la segunda sobreescribe el cache de la primera. En un servidor de producción con N empresas recibiendo mensajes simultáneos, esto causaría que una empresa respondiese con la personalidad de otra.

---

## Cambios en `config.js`

**Antes:**
```js
const { supabase, COMPANY_SLUG } = require('./clients');

let _configCache = null;
let _configCacheTime = 0;

async function obtenerConfigEmpresa() {
  // ...busca por slug = COMPANY_SLUG
}
```

**Después:**
```js
const { supabase } = require('./clients');

const _cache = new Map(); // Map<companyId, { data, cachedAt }>

async function obtenerConfigEmpresa(companyId) {
  // ...busca por id = companyId
}
```

La query cambia de:
```sql
WHERE slug = 'totalracks' AND estado = 'activo'
```
a:
```sql
WHERE id = '<companyId>' AND estado = 'activo'
```

El retorno `{ company, personality, knowledge }` es idéntico. Ningún consumidor necesita cambiar la forma en que usa el resultado.

---

## Impacto en otros módulos

| Módulo | Cambio requerido |
|---|---|
| `modules/orchestrator.js` | Llama `_obtenerConfig(message.company_id)` — se ajusta en T3.8 |
| `server.js` | Elimina import de `COMPANY_SLUG`; health/diagnostics usan nueva forma — se ajusta en T3.9 |
| `modules/clients.js` | No cambia. `COMPANY_SLUG` sigue exportándose pero ya no lo importa nadie después de T3.6+T3.9 |

## Prueba en aislamiento

```bash
node -e "
require('dotenv').config();
const { obtenerConfigEmpresa } = require('./modules/config');
// Obtener UUID de Total Racks primero desde Supabase
const UUID = '<uuid-totalracks>';
obtenerConfigEmpresa(UUID).then(r => {
  console.log('Empresa:', r.company.nombre);
  console.log('Asistente:', r.personality?.nombre_asistente);
  console.log('Knowledge secciones:', r.knowledge.length);
});
"
```

Resultado esperado:
```
Empresa: Total Racks
Asistente: TARA
Knowledge secciones: 5
```

Segunda llamada — debe venir del cache:
```bash
# Llamar dos veces en secuencia, la segunda debe tener latencia ~0ms
```

## Notas

El Orchestrator todavía llama a `this._obtenerConfig()` sin parámetros hasta que T3.8 lo actualice. Después de T3.6 y antes de T3.8, el sistema funcionará solo si se hace un ajuste temporal o si T3.6 y T3.8 se implementan en el mismo paso.

**Decisión:** T3.6 y T3.8 deben implementarse en la misma sesión de trabajo, sin deploy entre ellos, para evitar una ventana donde `config.js` requiere parámetro pero `orchestrator.js` no lo pasa.
