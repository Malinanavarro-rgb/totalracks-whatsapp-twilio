# T3.11 — Prueba con Segunda Empresa

**Estado:** Pendiente  
**Tipo:** Validación  
**Riesgo:** Ninguno — solo inserts de datos de prueba, sin cambio de código

---

## Objetivo

Antes de la validación completa de tres empresas (T3.12), confirmar que el mecanismo de routing funciona con una segunda empresa real. Este es el primer momento donde TARA atiende a dos empresas simultáneamente.

---

## Por qué antes de T3.12

T3.12 prueba tres empresas con personalidades cuidadosamente diseñadas para demostrar aislamiento. T3.11 es una prueba más simple: ¿funciona el routing básico con una segunda empresa? Si T3.11 falla, hay un bug de routing que resolver antes de escalar a tres empresas.

---

## Setup de la segunda empresa (SQL en Supabase)

La segunda empresa puede ser cualquier configuración simple. Ejemplo:

```sql
-- 1. Crear empresa
INSERT INTO companies (slug, nombre, descripcion, estado)
VALUES ('empresa-beta', 'Empresa Beta (Test)', 'Empresa de prueba para validar FASE 3', 'activo')
RETURNING id;
-- Guardar el UUID retornado

-- 2. Crear personalidad
INSERT INTO personalities (
  company_id, nombre_asistente, cargo, tono, objetivo,
  idioma, zona_horaria, modelo, temperatura, max_tokens,
  reglas, campos_requeridos
)
VALUES (
  '<UUID_EMPRESA_BETA>',
  'BETA', 'Asistente de Prueba',
  'neutro y directo',
  'Confirmar que el routing de FASE 3 funciona correctamente.',
  'es', 'America/Monterrey', 'gpt-4o-mini', 0.5, 200,
  '[{"texto": "Siempre presentarte como BETA, nunca como TARA.", "etapas": []}]',
  '{}'
);

-- 3. Crear knowledge base mínima
INSERT INTO knowledge_base (company_id, categoria, contenido)
VALUES (
  '<UUID_EMPRESA_BETA>',
  'Propósito',
  'Esta es la empresa Beta, utilizada únicamente para pruebas de routing de FASE 3. No tiene productos ni servicios reales.'
);

-- 4. Registrar un número de prueba
-- NOTA: Para sandbox de Twilio, un número diferente no existe físicamente.
-- La prueba se puede simular enviando manualmente una petición POST al webhook
-- con un campo To diferente, o usando un segundo número Twilio si se tiene.
INSERT INTO channel_endpoints (company_id, endpoint, canal, activo)
VALUES ('<UUID_EMPRESA_BETA>', 'whatsapp:+10000000001', 'whatsapp', true);
-- Este número es ficticio para la prueba; se reemplaza por uno real cuando exista
```

---

## Prueba de routing manual (sin número físico)

Si no hay un segundo número físico de Twilio disponible, la prueba se puede hacer directamente al endpoint:

```bash
# Simular mensaje al número de Empresa Beta
curl -X POST https://totalracks-whatsapp-twilio.onrender.com/webhook/twilio \
  -d "From=whatsapp:+52TEST123" \
  -d "To=whatsapp:+10000000001" \
  -d "Body=Hola" \
  -d "MessageSid=SMtest123" \
  -H "Content-Type: application/x-www-form-urlencoded"
```

**Nota:** La validación de firma de Twilio rechazará esta llamada si `WEBHOOK_URL_WHATSAPP` está definida. Para pruebas locales, comentar temporalmente la validación o usar una instancia local con ngrok.

---

## Qué verificar

1. **Routing correcto:** el mensaje al número de Empresa Beta activa la personalidad de BETA, no de TARA
2. **Aislamiento de config:** los logs muestran la config de Empresa Beta cargada (nombre, asistente)
3. **Aislamiento de datos:** en Supabase, el cliente creado tiene `company_id = UUID_EMPRESA_BETA`
4. **Total Racks sin afectación:** enviar mensaje al número de Total Racks → sigue respondiendo como TARA
5. **Número desconocido:** enviar a un número no registrado → TwiML vacío, sin crash

---

## Definition of Done

- [ ] Empresa Beta creada en Supabase (companies, personalities, knowledge_base, channel_endpoints)
- [ ] Routing verificado: número de Empresa Beta → personalidad BETA (no TARA)
- [ ] Total Racks verificado: número de Total Racks → personalidad TARA (sin cambio)
- [ ] Aislamiento de datos: `SELECT company_id FROM clientes ORDER BY created_at DESC LIMIT 5` muestra company_ids distintos para distintas empresas
- [ ] Número desconocido → TwiML vacío + log en Render, sin crash
- [ ] Commit realizado: `"test: segunda empresa validada — routing FASE 3 operativo"`

## Rollback

Eliminar los datos de Empresa Beta si se quiere limpiar:
```sql
DELETE FROM channel_endpoints WHERE company_id = '<UUID_EMPRESA_BETA>';
DELETE FROM knowledge_base      WHERE company_id = '<UUID_EMPRESA_BETA>';
DELETE FROM personalities       WHERE company_id = '<UUID_EMPRESA_BETA>';
DELETE FROM clientes            WHERE company_id = '<UUID_EMPRESA_BETA>';
DELETE FROM companies           WHERE slug = 'empresa-beta';
```
