# T3.7 — Refactor `crm.js` — Aislamiento por `company_id`

**Estado:** Pendiente  
**Tipo:** Código  
**Archivo:** `modules/crm.js`  
**Riesgo:** Medio — módulo que toca datos reales de producción. Un error aquí puede crear clientes duplicados o perder historial.

---

## Objetivo

Actualizar todas las funciones de CRM para que incluyan `company_id` en sus queries, garantizando que los datos de cada empresa nunca se mezclen.

---

## Por qué el CRM necesita cambiar

Hoy `obtenerOCrearCliente(telefono)` busca por teléfono en toda la tabla `clientes`, sin importar la empresa. En un sistema multi-tenant:

- Total Racks tiene un prospecto con número `+52 81 1234 5678`
- OLY NAILS también tiene una clienta con exactamente el mismo número
- Son personas distintas (o la misma persona en contextos distintos) con datos distintos

Sin `company_id` en el filtro, el cliente de Total Racks y el de OLY NAILS comparten la misma fila. El historial de conversaciones se mezcla. La etapa del pipeline se contamina.

---

## Funciones que cambian

### `obtenerOCrearCliente`
```js
// Antes:
async function obtenerOCrearCliente(telefono)

// Después:
async function obtenerOCrearCliente(telefono, companyId)
```

La búsqueda filtra por `(telefono, company_id)`. El INSERT incluye `company_id`.

### `guardarConversacion`
```js
// Antes:
async function guardarConversacion(clienteId, msg, resp, cat, int, sent)

// Después:
async function guardarConversacion(clienteId, companyId, msg, resp, cat, int, sent)
```

El INSERT en `conversaciones` incluye `company_id`.

### `crearOportunidadSiCorresponde`
```js
// Antes:
async function crearOportunidadSiCorresponde(clienteId, categoria, mensaje, intenciones)

// Después:
async function crearOportunidadSiCorresponde(clienteId, companyId, categoria, mensaje, intenciones)
```

El INSERT en `oportunidades` incluye `company_id`. La query de deduplicación filtra por `(cliente_id, company_id)`.

### `obtenerHistorial` y `actualizarScoreInteres`
No cambian — reciben `clienteId` que ya es específico de un cliente, y un cliente ya pertenece a una empresa después de T3.3.

---

## Impacto en el Orchestrator

El Orchestrator es quien llama a estas funciones. En T3.8 se actualizarán esas llamadas para pasar `message.company_id`. El CRM y el Orchestrator deben actualizarse en la misma sesión (igual que T3.6 + T3.8).

---

Requiere haber completado T3.3 (columnas existen) y T3.6 (para tener `companyId` disponible).

```bash
node -e "
require('dotenv').config();
const { obtenerOCrearCliente } = require('./modules/crm');
const UUID = '<uuid-totalracks>';
obtenerOCrearCliente('+52000000000', UUID).then(r => {
  console.log('Cliente:', r?.id, r?.telefono, r?.company_id);
});
"
```

Resultado esperado: cliente creado (o encontrado si ya existe) con `company_id = UUID_TOTALRACKS`.

```bash
# → deben ser dos filas separadas en la tabla clientes
```
